05 可观测与迁移：可用到可运营

结构化日志、回放链路、分阶段迁移与回滚验收。

05可观测与迁移

本章目标：把系统从“能跑”提升到“可运营”，做到可定位、可修复、可验收不回归。

01记忆系统 02主循环 03工具编排 04多 Agent 05可观测迁移

1. 没有可观测，你会遇到什么

典型场景：

  • 用户说“卡住了”，你只能看到最后一条错误。
  • 你知道失败了，但不知道在哪个转移点失败。
  • 你修了一个点，却引入另一个隐蔽回归。

问题根因通常不是“没日志”，而是“日志不结构化”。

2. 最小可观测字段集合（必须有）

建议至少包含：

  1. state_from
  2. state_to
  3. transition_reason
  4. error_class
  5. recovery_attempt
  6. terminal_reason
  7. memory_injection_bytes
  8. tool_result_bytes

这 8 个字段足够支撑：

  • 状态机回放。
  • 恢复链失败定位。
  • 上下文与工具体积问题定位。

4. 迁移路线：从 0 到 1 再到可运营

建议分 4 阶段推进，每阶段都可验收、可回滚。

阶段 A（2-5 天）：先可控

目标：

  1. 完成基础工具链（add/replace/remove、注入、预算）。
  2. 关键转移点有结构化日志。

验收：

  • 主流程能跑通。
  • 问题可定位到具体转移节点。

阶段 B（1-2 周）：再稳态

目标：

  1. 回合抽取与去重。
  2. 恢复链分型和计数。

验收：

  • 常见故障可自动恢复。
  • 恢复失败有明确终止语义。

阶段 C（1-3 周）：再规模化

目标：

  1. 索引治理与 fallback。
  2. 多会话/多代理治理。

验收：

  • 负载上来后不失稳。
  • 权限边界和资源回收可验证。

阶段 D（持续）：持续优化

目标：

  1. 周期整理、漂移校正。
  2. 指标驱动迭代。

验收：

  • 核心指标持续改善，而不是“偶尔成功”。

5. 为什么每阶段都必须“可回滚”

AI 系统的不确定性比传统业务逻辑高。

如果没有回滚边界：

  • 一次上线可能同时影响提示词、上下文、工具链、策略。
  • 你会在故障时无法判断根因层级。

建议：

  • 每阶段只改一层核心变量。
  • 上线前写好回滚条件。

6. 指标如何选：别贪多，先抓关键

建议先抓 6 个：

  1. task_success_rate
  2. recovery_success_rate
  3. terminal_rate
  4. avg_turn_latency_ms
  5. context_overflow_rate
  6. tool_failure_propagation_rate

这些指标可以直接回答：

  • 系统到底稳不稳。
  • 修复到底有没有真实收益。

7. 风险与治理

风险一：日志很多但不可用

没有统一 schema 的日志，通常只能“看起来很多”。

治理：

  • 先定字段，再扩指标。

风险二：迁移节奏过快

一次改太多层，故障定位几乎不可能。

治理：

  • 分阶段灰度。
  • 每阶段单独验收。

风险三：只看成功率

成功率可能上升，但成本和时延也可能失控。

治理：

  • 成功率、时延、成本、失败传播一起看。

8. 学习这一页后，你应该能回答

  1. 为什么“可观测”是 Agent 工程的地基。
  2. 为什么迁移必须阶段化和可回滚。
  3. 为什么指标体系要覆盖成功、恢复、失败、成本四个维度。
