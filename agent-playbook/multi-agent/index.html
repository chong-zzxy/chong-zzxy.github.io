04 多 Agent 治理：放权不失控

多 Agent 的 spawn gate、权限收缩、结果回收与统一清理。

1. 先定一个原则：子代理不是自由体

子代理应该是“受约束的执行单元”，不是“复制一个主代理再放出去跑”。

从工程角度，至少要回答：

  1. 谁允许它被创建。
  2. 它最多能拿到什么权限。
  3. 它完成后如何回收到主链路。
  4. 出错后如何清理资源。

2. 项目里可借鉴的治理思路

结合主循环和相关模块，可提炼出四段式治理链：

  1. Spawn Gate：深度、并发、allowlist、上下文模式。
  2. Capability Shrink：父能力与任务需求做交集。
  3. Deterministic Merge：执行可并发，回放需有序。
  4. Unified Cleanup：所有失败路径统一清理。

这四段缺一不可。

4. 结果回收：为什么必须顺序稳定

多子任务并发执行后，如果直接按完成时间输出，会导致：

  1. 结果顺序不稳定。
  2. 同输入多次运行难复现。
  3. 人类审阅成本大幅上升。

建议：

  • 执行层并发。
  • 展示层按固定 key（如 task_index）排序回放。

5. 失败清理：为什么要统一 checklist

多 Agent 最大的隐性成本往往不是失败本身，而是失败后的“残留”：

  1. 子会话没清理。
  2. 进程没回收。
  3. 临时文件或附件没删。
  4. 锁没释放。

建议把清理做成一个强制 finally 流程。

6. 为什么这套设计能抗“规模化失控”

6.1 Spawn Gate 控规模

没有 gate 的并发，本质是“无限递归创建任务”。

6.2 Capability Shrink 控风险

权限收缩保证每一层子代理都在更小的安全边界内运行。

6.3 Deterministic Merge 控可复盘性

可复盘是工程质量的基础，尤其在多代理系统里。

6.4 Unified Cleanup 控系统健康

清理一致性决定长期稳定性。

7. 风险与取舍

风险一：策略过严导致效率下降

治理过严会抑制并发收益。

建议：

  • 先保安全，再逐步放宽。

风险二：策略过松导致越权

一旦放宽默认权限，问题会在规模扩大时爆发。

建议：

  • 默认拒绝，按需放行。

风险三：只治理权限，不治理回收

结果是“权限看起来安全，但系统资源慢性泄漏”。

建议：

  • 权限治理与清理治理一起设计、一起验收。

8. 学习这一页后，你应该能回答

  1. 为什么多 Agent 的第一问题是治理，而不是并发数量。
  2. 为什么子代理能力必须收缩而非继承。
  3. 为什么回放顺序稳定性和清理一致性决定可运营性。
