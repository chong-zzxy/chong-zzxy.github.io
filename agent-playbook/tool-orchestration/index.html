1. 工具编排的三大问题

  1. 什么能并发，什么不能并发。
  2. 一个工具失败后，影响范围怎么控制。
  3. 工具结果太大时，怎么避免拖垮上下文。

这三件事如果不提前设计，后面只能靠“临时 if”救火。

2. 项目里的关键思路

从 src/query.ts 和工具编排相关模块可以提炼出三条高价值实践。

2.1 并发判定前置化

先判断冲突，再决定并发。

冲突通常来自：

  1. 文件路径写冲突。
  2. 状态依赖冲突。
  3. 顺序语义冲突（后一步依赖前一步输出）。

2.2 并发执行，顺序回放

这是很多系统忽略但很关键的一点：

  • 执行可以并发拿性能。
  • 对模型和用户的结果回放必须稳定顺序。

否则每次同样输入，输出叙事顺序都不同，排障会极其困难。

2.3 工具结果预算器前置

在主循环里（applyToolResultBudget）先限制超大结果，再进后续压缩链。

意义：

  • 在“发送给模型之前”就减负。
  • 避免把巨大文本都塞进上下文后才补救。

4. 失败传播怎么设计才不“全挂”

失败传播建议做成工具级策略，而不是写死全局策略。

例如：

  1. local_fail：单工具失败，其他工具继续。
  2. sibling_abort：同批兄弟调用中止（例如共享资源型工具）。
  3. global_stop：整轮停止（高风险破坏性工具）。

6. 为什么“并发执行 + 顺序回放”特别重要

你可以把它理解成：

  • 后台是乱序计算。
  • 前台是有序叙事。

如果前台也乱序，三个后果几乎必然出现：

  1. 用户理解成本急剧上升。
  2. 复盘时难以重建真实路径。
  3. 评测数据不稳定，A/B 结论失真。

7. 风险与边界

风险一：并发判定过于粗糙

只按“工具名”判定并发，忽略路径和依赖，会造成隐性冲突。

风险二：失败策略过于乐观

默认“都局部失败”会让关键失败被悄悄吞掉。

风险三：超大结果只做后置处理

如果不做前置预算，模型前的上下文就已经爆了，后置兜底来不及。

  3. 为什么工具结果治理要双层防线。
