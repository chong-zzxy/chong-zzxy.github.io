02主循环

本章目标：把主循环做成可恢复、可终止、可解释的状态机，重点控制状态转移、恢复限额和失败语义。

1. 主循环不是 while true，而是状态机

在 src/query.ts 里你能看到，循环虽然是 while (true)，但真实控制逻辑是“状态 + 转移原因”。

这个思路非常关键：

  • while 是语法壳。
  • state 才是行为核心。

为什么必须这么做

如果你只靠 if-else 堆分支，最终会发生：

  1. 某个恢复路径被重复触发。
  2. 多个恢复路径互相覆盖。
  3. 系统“看起来还在跑”，但没有明确进展。

2. 项目里的恢复链设计（你可以直接借鉴）

2.1 上下文溢出恢复

在 query.ts 里有多层恢复：

  1. 工具结果预算裁剪（先减体积）。
  2. 历史 snip / microcompact / collapse（再压上下文）。
  3. autocompact（必要时摘要压缩）。
  4. 失败时进入明确终止。

为什么顺序重要：

  • 先做低损恢复，再做高损恢复。
  • 避免“一上来就重压缩”导致语义损失过大。

2.2 max_output_tokens 恢复

代码里有独立恢复计数（如 MAX_OUTPUT_TOKENS_RECOVERY_LIMIT）。

这体现一个优秀实践：

  • 输出截断问题与上下文溢出问题分开治理。
  • 分开计数，分开终止条件，避免互相污染。

2.3 stop hooks 的后台化

src/query/stopHooks.ts 里把提示建议、记忆提取、autoDream 等放到 stop hooks 阶段。

这有两个好处：

  1. 主链路保持响应优先。
  2. 背景任务失败不会拖垮主对话。

3. 主循环设计中的三个“硬原则”

3.1 每次转移必须有 reason

没有 reason，就没有可观测性；没有可观测性，就无法稳定迭代。

3.2 每个恢复分支都要限额

否则单点故障会变成无限重试风暴。

3.3 一定要有 terminal

“尽量恢复”不等于“永不失败”。

真正可靠的系统，是可以体面失败并给出可行动信息的系统。

5. 为什么这些恢复链能解决真实问题

5.1 分层恢复防止语义崩塌

如果每次都直接摘要压缩，系统短期看“恢复了”，长期会因为语义损失累积而变得越来越笨。

5.2 独立计数防止串扰

overflow 和 output truncation 分开计数，可以避免：

  • 你以为在恢复 A，实际上在无限重试 B。

5.3 stop hooks 异步化防止主链路阻塞

让“锦上添花”不阻塞“雪中送炭”。

6. 风险与边界

风险一：恢复成功但事实丢失

压缩永远有信息损耗风险。

建议：

  • 对关键事实设置“不可压缩片段”或“高优先级保留”。

风险二：计数器太保守

恢复次数设太低，系统会“过早放弃”。

建议：

  • 按故障类型分别调参，不要统一一个魔法数字。

风险三：只记录失败，不记录转移

这会导致你只能看到 crash，无法看到 crash 前的路径。

建议：

  • 把 state_from/state_to/reason 作为一等日志字段。
