驾驭工程（Harness Engineering）：让 Agent 稳定交付

Harness Engineering 的核心不是“更聪明的模型”，而是“更稳定的执行环境与反馈闭环”。

2026年04月14日 更新于 2026年04月14日 Agent工程 约 3 分钟阅读

OpenAI 对 Harness Engineering 的一句话非常到位：
Humans steer. Agents execute.

如果把提示词工程理解成“怎么下指令”，把上下文工程理解成“喂什么信息”，那 Harness Engineering 就是：

如何把整个执行系统做成可控、可复盘、可持续优化的工程闭环。

1. 什么是 Harness Engineering

Harness 直译是“驾驭装置”。在 Agent 工程里，它不是某个工具，而是一整套基础设施：

  1. 规则文件（行为边界）。
  2. 工具系统（执行能力）。
  3. 验证系统（质量把关）。
  4. 观测系统（问题定位）。
  5. 反馈系统（持续迭代）。

你可以把它理解为“让代理可运营”的骨架。

2. 为什么很多 Agent 项目止步于 demo

常见原因不是模型不够强，而是 harness 不完整：

  1. 没有统一约束，代理行为漂移。
  2. 没有验证门，修复和回归混在一起。
  3. 没有观测链，线上问题无法复盘。
  4. 没有反馈闭环，每次都靠人工经验重来。

因此，Harness Engineering 的重点是“系统行为稳定性”，不是“单次回答质量”。

3. 从项目抽象出来的 Harness 结构

结合项目中的运行逻辑，可以抽象出五层 harness。

3.1 Policy Layer（策略层）

  • 定义代理边界、权限模式、失败处理原则。
  • 典型载体：AGENTS.md、权限策略、工具 allowlist。

3.2 Runtime Layer（运行层）

  • 负责 query loop、工具调度、恢复链执行。
  • 核心目标：稳定推进、可终止。

3.3 Verification Layer（验证层）

  • 代码改动后自动执行测试、lint、关键检查。
  • 禁止“未验证即宣称成功”。

3.4 Observability Layer（可观测层）

  • 记录状态转移、错误分类、恢复次数、终止原因。
  • 支持线上回放和指标聚合。

3.5 Learning Layer（学习层）

  • 把故障复盘沉淀为新规则、技能、评测样本。
  • 实现“失败一次，系统变强一次”。

4. Harness 的核心闭环（Python，带注释）

这段代码最重要的点不是语法，而是顺序：

  • 没有验证，不算完成。
  • 没有记录，不算可运营。
  • 没有学习，不算工程化。

5. 如何落地一个“最小可用 Harness”

你可以先从下面 5 个文件/机制开始。

  1. AGENTS.md：明确代理行为边界与输出规范。
  2. skills/：把重复流程做成可复用技能。
  3. eval/：沉淀失败样本，做回归评测。
  4. scripts/verify.*：统一验证入口。
  5. logs/structured：统一日志字段。

这就是一个最小骨架，已经能把 demo 变成可迭代系统。

6. Harness Engineering 能解决什么问题

问题一：输出风格漂移

通过策略层和模板约束，减少随机波动。

问题二：修复质量不稳定

通过验证层和评测集，减少“看似修复”。

问题三：线上问题复盘困难

通过可观测层，建立状态转移与故障因果链。

问题四：团队经验无法沉淀

通过学习层，把个体经验升级为系统能力。

7. 风险与取舍

风险一：harness 过重

流程太多会拖慢迭代速度。

建议：

  • 先做最小骨架，再按故障频率扩展。

风险二：只做规范，不做执行

文档写得很好，但运行时不 enforce，等于没有 harness。

建议：

  • 能自动检查的都自动检查。

风险三：只记录，不分析

日志堆积但不形成反馈，系统不会变强。

建议：

  • 每周固定做故障样本回灌。

8. 你可以直接拿走的 Harness 迭代节奏

第 1 周

  • 补齐策略层与验证层最小闭环。

第 2 周

  • 补齐可观测字段，完成状态转移回放。

第 3 周

  • 建失败样本库和回归评测。

第 4 周及以后

  • 以指标驱动优化，持续沉淀技能。

9. 结语

Harness Engineering 的价值，不在于让代理“偶尔惊艳”，而在于让系统“长期稳定”。

当你把“规则、执行、验证、观测、学习”连成闭环时，Agent 才真正从玩具变成生产力。
