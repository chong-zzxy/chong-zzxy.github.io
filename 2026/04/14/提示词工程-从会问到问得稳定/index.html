提示词工程：从“会问”到“问得稳定”

提示词工程不是“咒语学”，而是把目标、约束、验收标准结构化表达，并通过评测持续迭代。

2026年04月14日 更新于 2026年04月14日 Agent工程 约 4 分钟阅读

提示词工程最容易被误解成“技巧集合”，比如某个万能开头、某个神奇词。

更接近真相的定义是：提示词工程是把意图翻译成可执行规范的过程。
你给模型的不是“灵感”，而是“任务合同”。

1. 什么是提示词工程

一句话定义：

提示词工程 = 任务定义 + 约束定义 + 输出定义 + 验收定义。

如果只写“帮我总结一下”，通常会得到“可读但不可用”的结果。
如果写成“目标、输入边界、输出格式、失败处理、验收标准”，稳定性会明显提高。

2. 多平台共识：提示词工程在做什么

下面是我查阅几个平台官方资料后提炼的共同点。

平台 共同强调点 工程含义
OpenAI（Codex/Prompting） 目标与约束要明确、可验证 提示词是规范，不是聊天
Anthropic（Prompt Engineering） 给清晰指令、结构化上下文、示例驱动 先定义任务边界，再谈润色
Google（Gemini Prompt Design） 角色、上下文、期望输出需要显式表达 好提示词是可复现模板

共同结论：

  1. 先写任务合同，再写语言修饰。
  2. 输出格式要先定，否则无法稳定自动化。
  3. 评测数据集要先建，否则无法判断是否真的变好。

你可以把任何提示词先写成这 5 段。

这五段不是“形式主义”，它让模型从“自由发挥”切到“按规格交付”。

4. 四类高频任务的写法

4.1 生成类任务

目标是让模型写东西，但你要控制风格、长度、结构和事实边界。

4.2 抽取类任务

核心是结构化和可追溯。

4.3 规划类任务

先要求“计划”，再要求“执行”，可以大幅降低混乱输出。

4.4 评审类任务

如果你不写评审标准，模型只会给泛泛建议。

5. 提示词工程不是一次写完，而是评测驱动迭代

一个常见误区：

“这个提示词看起来不错”，不等于“线上稳定”。

你需要一个最小评测闭环。

建议最少追四个指标：

  1. task_pass_rate：任务达成率。
  2. format_error_rate：格式错误率。
  3. hallucination_rate：幻觉率。
  4. latency_cost：时延与成本。

6. 常见失败模式与修复

失败模式 典型症状 修复方式
目标不清 输出方向漂移 把目标写成可验收句子
约束缺失 输出不合规 明确禁止项与失败处理
格式未定义 结果难自动化处理 强制 JSON schema
一次塞太多任务 输出前后矛盾 拆成“规划-执行-校验”三阶段
无评测 看起来变好但线上变差 建立固定 eval set

7. 给 Agent 场景的专用建议

当你把模型当“代理”而不是“问答器”时，提示词要从“生成导向”升级为“执行导向”。

执行导向提示词至少要包含：

  1. 权限边界：能做什么，不能做什么。
  2. 工具策略：什么时候用工具，失败如何回退。
  3. 验证要求：改动后必须跑哪些检查。
  4. 报告格式：输出要让人类快速决策。

9. 结语

提示词工程真正的进阶，不是“更会说话”，而是“更会定义工作”。

当你把提示词写成可执行规范，模型输出就会从“有时惊艳”变成“稳定可用”。
