上下文工程：把“喂信息”做成系统能力

上下文工程不是“塞更多信息”，而是让模型在正确时间看到正确信息，并可验证、可回放、可治理。

2026年04月14日 更新于 2026年04月14日 Agent工程 约 3 分钟阅读

很多团队做 Agent 时，最先遇到的问题不是推理能力，而是上下文质量：

  1. 给少了，模型答不对。
  2. 给多了，模型变慢还容易跑偏。
  3. 给杂了，模型抓不住重点。

这就是上下文工程要解决的核心：
在正确时间，把正确信息，以正确结构喂给模型。

1. 上下文工程是什么

一句话：

上下文工程 = 信息选择 + 信息排序 + 信息压缩 + 信息治理。

它不是“RAG 的同义词”，而是更大的一层系统工程。

RAG 负责“找信息”，上下文工程负责“让信息可用且可控”。

2. 为什么它比提示词更难

提示词通常是静态文本。
上下文是动态系统，至少涉及五个变量：

  1. 数据源是否可信。
  2. 检索是否相关。
  3. 注入是否超预算。
  4. 历史是否过期。
  5. 压缩是否保真。

你可以把提示词理解为“任务合同”，把上下文工程理解为“合同执行现场”。

3. 从项目里提炼出来的上下文设计逻辑

基于当前项目代码链路，可以提炼四个关键设计决策。

3.1 预取前置且非阻塞

query.ts 里的 startRelevantMemoryPrefetch(...) 说明一个思路：

  • 尽早预取。
  • 不阻塞主链路。
  • 结果可用就注入，不可用就跳过。

解决的问题：

  • 首 token 延迟不被记忆检索拖垮。

3.2 注入前先做预算治理

applyToolResultBudget 在主循环较早阶段执行，先处理超大工具结果。

解决的问题：

  • 避免“把大块垃圾塞进上下文后再补救”。

3.3 压缩链分层

可见多层压缩策略（snip/microcompact/collapse/autocompact）并非重复，而是分损耗等级。

解决的问题：

  • 尽量先做低损压缩，避免语义一刀切。

3.4 回合后沉淀

stopHooks.ts 把 extractMemories、autoDream 放在回合后异步执行。

解决的问题：

  • “记忆沉淀”不阻塞“当前回答”。

4. 一个实用的上下文分层模型

推荐把上下文拆成四层：

  1. Layer-0 任务必需层（system/developer 规则、当前任务目标）。
  2. Layer-1 当前回合层（用户输入、最近关键交互）。
  3. Layer-2 外部证据层（检索文档、工具输出、结构化事实）。
  4. Layer-3 长期记忆层（偏好、历史决策、组织约束）。

注入顺序建议：

  • 先 L0/L1（保证任务不跑偏）
  • 再 L2（补事实）
  • 最后 L3（补长期偏好）

6. 上下文工程里的常见误区

误区 后果 修正
只追召回率，不追注入质量 相关但无用信息挤占预算 加注入后指标（命中贡献率）
只做检索，不做压缩 长上下文高成本低收益 低损压缩优先，分层降损
只记住，不遗忘 历史污染持续累积 replace/remove/过期机制一等化
把工具结果全量注入 上下文爆炸、性能下降 预算器前置 + 结构化摘要

7. 如何判断你的上下文工程做对了

建议看四类指标：

  1. context_relevance_rate：注入内容对答案有贡献的比例。
  2. context_waste_tokens：注入但未被利用的 token。
  3. overflow_recovery_rate：上下文溢出恢复成功率。
  4. memory_staleness_rate：命中记忆里过期项比例。

如果你只看任务成功率，你会错过成本和稳定性问题。

8. 风险与边界

风险一：检索正确但排序错误

信息都找到了，但关键片段被排到后面，模型先看到了噪声。

风险二：压缩成功但语义丢失

尤其在高损压缩阶段，最容易丢“否定条件”和“边界约束”。

风险三：长期记忆污染

一次错误确认会在后续回合持续放大。

9. 结语

提示词决定你“想让模型做什么”。

上下文工程决定模型“实际能做到什么”。

当上下文工程做扎实后，模型能力的上限才真正能被释放出来。
